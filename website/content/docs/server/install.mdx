# Derrick Server Install

The recommended way to install the server is using the
[`derrick install`](/waypoint/commands/install) command.
This command will install a server into Docker, Kubernetes, AWS ECS, or Nomad,
bootstrap the server, and configure your local CLI to access that server.
It is a single command to get up and running with Derrick.

This command is configured to install recommended defaults out of the box, while
giving enough dials and options for users to tweak to still be useful. If there's
a specific setup that doesn't work for you with `derrick install`, we recommend
looking at the [server run](/waypoint/docs/server/run) documentation instead or opening
an issue on GitHub and requesting a feature enhancement.

The server installed by the `derrick install` command runs Derrick
server in a Docker container, which runs the `derrick server run`
command:

```shell-session
derrick server run -accept-tos -vv -db=/data/data.db -listen-grpc=0.0.0.0:9701 -listen-http=0.0.0.0:9702
```

## Officially Supported Server Platforms

Currently, Derrick supports running its server on the following platforms:

- [Kubernetes](/waypoint/docs/server/install#kubernetes-platform)
- [Nomad](/waypoint/docs/server/install#nomad-platform)
- [Amazon ECS](/waypoint/docs/server/install#aws-ecs-platform)
- [Docker](/waypoint/docs/server/install#docker-platform)

### Kubernetes Platform

Installing the Derrick server to Kubernetes using the `derrick install` command
uses the [hashicorp/waypoint-helm](https://github.com/hashicorp/waypoint-helm)
chart. Derrick install will set sensible defaults for many of the Helm chart's
values, while using flags for customization of some values as well.

You may also use the Helm chart directly and install Derrick using `helm install`.
We recommend reading the README detailed on the derrick-helm project for more
information on how to further customize your Derrick installation to Kubernetes
with Helm.

### Nomad Platform

To install Derrick server on Nomad, there are a few requirements that should be
met prior to running the install:

You must use a CSI or Host volume to store Derrick server's database. A host volume
or CSI volume provider can be specified as a command line option to `derrick install` and
should be registered in Nomad prior to running the `derrick install` command.

This command sets up a Nomad job for running Derrick server and its
runner as two separate jobs by default.

~> **NOTE**: **We strongly recommend using Consul** for networking support in Nomad running Derrick server.

By default, the CLI installer will automatically create a service for Consul in
Nomad for running Derrick server. This is an important default, as it gives
Derrick server a stable hostname for your clients to point at, or remote
runners to use for calling back to the Derrick server. There are a handful of
installer flags that let you configure what tags the Derrick server and UI
service should have, what datacenter or domain Consul is in, or
define your own Consul service hostname to use. By default, the
`derrick install` helper will return a hostname crafted using the default
expected host for a Consul service, i.e. `derrick-server.service.dc1.consul`.

When running the install helper, the CLI should show the Consul DNS backed hostname of Derrick
server at the end of an install:

```shell-session
$ derrick install -platform=nomad -nomad-host-volume="waypoint" -nomad-runner-host-volume="waypoint-runner" -accept-tos
✓ Derrick server ready
✓ The CLI has been configured to automatically install a Consul service for
the Derrick service backend and ui service in Nomad.
✓ Server installed and configured!
✓ Successfully connected to Derrick server in Nomad!
✓ Installing runner...
✓ Derrick runner installed
Derrick server successfully installed and configured!

The CLI has been configured to connect to the server automatically. This
connection information is saved in the CLI context named "install-1635801894".
Use the "derrick context" CLI to manage CLI contexts.

The server has been configured to advertise the following address for
entrypoint communications. This must be a reachable address for all your
deployments. If this is incorrect, manually set it using the CLI command
"derrick server config-set".

To launch and authenticate into the Web UI, run:
derrick ui -authenticate

Advertise Address: derrick-server.service.dc1.consul:9701
Web UI Address: https://waypoint-ui.service.dc1.consul:9702
```

Notice that the advertise and web ui address is shown as a hostname, rather than an IP.

Without using Consul, if the Derrick server allocation ever gets restarted in Nomad, the IP
address will no longer be valid, and you will have to upgrade every server client's context.
There are ways around this issue that are considered more advanced, and are likely
better candidates for using the [server run](/waypoint/docs/server/run) flow for setting
up Derrick server.

If you do not wish to automatically register a Consul service in Nomad for
Derrick server by default with `derrick install`, it can always be disabled
by including the `-nomad-service-provider=none` flag. You should then see
an IP address from the Nomad allocation after the Derrick install helper finishes.
You may also use Nomad as the service discovery provider with `-nomad-service-provider=nomad`.

If your Nomad cluster is ACL-enabled, you will need the `NOMAD_TOKEN` environment variable
set with a token that has permissions for submitting a job to the specified namespace.
Likewise, if you are installing the Derrick server and using Consul for service
discovery to an ACL-enabled Consul cluster, you may need to set the environment
variable `CONSUL_HTTP_TOKEN`, depending on your Nomad client configuration.

### AWS ECS Platform

Derrick supports installing the server in AWS ECS. Several AWS resources are created
as part of the install. The default cluster name is `derrick-server`; this can
be changed by specifying the value for the flag `-ecs-cluster`. The resources
are tagged with `derrick-server` as the tag name; this is not able to be changed
as Derrick needs a way to track all the resources that are part of the server install.

The main components installed are:

- EFS Filesystem: used for persisting the Derrick server database
- EC2 Security Group: authorizing traffic in to the cluster, as well as enabling access to the EFS FileSystem from the Cluster
- CloudWatch Log Groups: a log group is created for each of the Derrick server and runner services
- IAM Execution role
- Network load balancer: required to enable both HTTP and gRPC connections (see `NOTE`)
- Load balancing target groups: a target group is created for each of the Derrick server and runner services
- ECS Cluster: a cluster dedicated to running the Derrick server and runner
- ECS server service: runs the Derrick server task
- ECS runner service: runs the default Derrick runner task
- ECS Task Definitions: for both the server and runner

~> **NOTE**: A Network Load Balancer is required because user-supplied TLS certificates are not yet supported.

An example ECS install should result in output similar to the below:

### Docker Platform

Derrick supports running the Derrick server directly in Docker. Generally, this flow
is recommended only for development to get started using Derrick. We do not
recommend using Derrick running in Docker for production workflows.

This command sets up Docker containers for running Derrick server and its
runner as two separate containers by default. They are each installed to a Docker
network called "waypoint".
