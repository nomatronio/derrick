# Project Configuration: `derrick.hcl`

The `derrick.hcl` file is used to configure each Derrick project. It defines
how Derrick builds, deploys, and releases a project. The `derrick.hcl` file is
written in [HCL](/waypoint/docs/derrick-hcl/syntax) or [JSON](/waypoint/docs/derrick-hcl/syntax/json),
allowing both human and machine friendly formats.

## `derrick.hcl` Location

Derrick expects one `derrick.hcl` per [project](/waypoint/docs/projects).
The `derrick.hcl` configuration can be a file committed alongside your
project source, or it can be stored on the Derrick server without modifying
your version controlled source at all.

For any Derrick operation such as `derrick up`, Derrick will always prefer
a local `derrick.hcl` file if it is found, even if the project is associated
with a server-side configuration.

### File

The `derrick.hcl` file may be committed as a file
to version control. This allows the Derrick behavior of a project to be
versioned directly alongside the application. The downside of this approach
is that it requires making changes to your project source.

For this approach, the `derrick.hcl` file is typically placed at the root
of a version control repository. To deploy multiple applications, define each
with an [`app`][app] stanza.

When executing the `derrick` CLI, it will search for the `derrick.hcl` file
in the current directory, followed by each subsequent parent directory.

### Server

The `derrick.hcl` configuration may be stored on the Derrick server
and associated directly with a [project](/waypoint/docs/projects).

The benefit of this approach is that no changes to project source repositories
are required and the team managing the Derrick configuration and the team
_using_ Derrick can be easily separated.

This approach also makes it very easy to migrate existing applications to
Derrick, perhaps using a GitOps-style deployment pattern, since you can
create and configure the projects completely in the web UI without having
to touch any existing source code.

To store the `derrick.hcl` configuration on the Derrick server, specify
the `derrick.hcl` contents in the web UI when prompted or specify the
`-derrick-hcl` flag to the [`derrick project apply`](/waypoint/commands/project-apply)
command.

## Template `derrick.hcl`

```hcl
project = "project-name"

app "app-name" {
  path = "./src"

  build {
    # ...
  }

  deploy {
    # ...
  }

  release {
    # ...
  }

plugin "plugin-name" {

}
```

## Top-level

The "top-level" refers to the objects that are not nested in any other
configuration. The top-level is made of "stanzas" and "parameters".

## Stanzas and Parameters

A "stanza" is a block of code that opens and closes with braces `{ }`
and may contain more configuration parameters. Above, `app "app-name" {}` is
the `app` stanza.

A "parameter" is the term used to describe any configuration object. `build`
and `path` are parameters of `app`. `project`, `app "app-name"`, and `plugin`
are parameters of the top-level.

Several "common" parameters can be used in stanzas. These are `use` and `hook`.
These can not be top-level parameters, but are available in `build`, `deploy`,
and `release`. These are documented further in their respective pages.

A stanza is a parameter, but not all parameters are stanzas.

## Variables

Derrick configuration files have access to a number of predefined variables.
Using variables helps make a Derrick configuration more robust by more
explicitly specifying paths, reducing duplication, etc. You can learn more about
variables in the dedicated [variables section](/waypoint/docs/derrick-hcl/variables).

## Top-level Parameters

### Required

- `app` <code>([app][app])</code> - One or more applications to build, deploy,
  and release with Derrick.

- `project` `(string)` - The name of the project. This should be unique
  for the Derrick server and must not be changed later. This is used to
  organize information on the server. **This field is optional if you're
  using [remote operations](/waypoint/docs/projects/remote), including those
  triggered via [Git polling](/waypoint/docs/projects/git).**

~> **Warning: the project name can not be changed.** Changing the project
name is not currently supported. If you change the name, the history,
deployments, etc. of the previous project name will not be migrated or destroyed.
A future version of Derrick will enable renaming projects.

### Optional

- `plugin` <code>([plugin][plugin])</code> - External plugins that may be
  used in this configuration. Plugins are implicitly defined with
  [`use`](/waypoint/docs/derrick-hcl/use) stanzas so this is only required if you
  need to additionally configure a plugin.
